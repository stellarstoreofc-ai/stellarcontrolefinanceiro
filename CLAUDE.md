# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

**Nexus Control** is a single-file financial tracker for the business's own accounts payable and expenses ("contas a pagar" / "contas que saíram"). It is a private internal tool (not a public SaaS) but is backed by Supabase for auth + storage so the same data is available from any device after logging in — no per-browser `localStorage` lock-in.

## Running / Previewing

Open `index.html` directly in a browser. No server, build step, or package manager. Requires network access to reach Supabase (both for auth and data).

## Deployment

Deployed via Vercel, connected to this GitHub repo (auto-deploys on push to `main`) — live at `https://nexuscontrols.vercel.app`. Pure static HTML, no build command needed. Netlify drag-and-drop (app.netlify.com) also works as a one-off alternative if needed. The Supabase URL/anon key are hardcoded in `index.html` (see below) — no environment variables or build-time injection.

If the deployed URL ever changes, update it in Supabase under Authentication → URL Configuration (Site URL / Redirect URLs) — otherwise signup confirmation emails will link to the wrong place.

## Supabase setup (one-time, per project)

Run `supabase-setup.sql` in the Supabase SQL Editor to create the `financeiro_dados` table and its Row Level Security policies before first use. See `README.md` for the full first-run checklist (creating the login user, email confirmation). That file's header comments also list `alter table` statements for columns (`custo_fixo`, `produtos`, `config_precificacao`) added after the table's initial creation — run those against any pre-existing table that predates a given column.

**Never put the `service_role` key in this file or anywhere client-side** — only the `anon` key belongs in `index.html`. Security comes entirely from Supabase Auth + the RLS policies (`auth.uid() = user_id`) in `supabase-setup.sql`, not from hiding the anon key.

## Architecture

Single file (`index.html`) containing markup, `<style>`, and `<script>`. No frameworks; the only external dependency is `@supabase/supabase-js@2` loaded via CDN `<script>` tag.

The app has three top-level `.screen` divs toggled by `showScreen(id)`: **`screen-login`** (email/senha form, doubles as signup via the "Criar conta" toggle, plus an "Esqueci minha senha" button that calls `sb.auth.resetPasswordForEmail(email, { redirectTo: <this same URL> })`), **`screen-reset`** (a "definir nova senha" form, see below), and **`screen-app`** (the sidebar SPA described below). On load, `initAuth()` checks `sb.auth.getSession()` — an existing session (persisted by supabase-js in its own `localStorage` entry) skips straight to the app; otherwise the login screen shows. `afterLogin(user)` sets `currentUserId`, calls `carregarDados(user.id)` to pull this user's row, then shows the app.

**Password-recovery link handling**: a recovery email's link lands back on this same page with `#access_token=...&type=recovery` in the URL hash. supabase-js's `detectSessionInUrl` auto-authenticates that as a normal session, so without special-casing it `initAuth()` would send the user straight into the app instead of letting them set a new password. `linkEhRecoverySenha` (`/type=recovery/.test(window.location.hash)`, checked once at script load, before any auth call) is what `initAuth()` uses to route to `screen-reset` instead of `afterLogin()` when both a session *and* that hash are present. `formResetSenha`'s submit handler calls `sb.auth.updateUser({ password })`, strips the token from the URL via `history.replaceState` (so a refresh can't reuse it and it's not left sitting in the address bar), then calls `afterLogin()` directly — no re-login needed, the session from the recovery link is already valid. `btnCancelarReset` signs out and clears the hash instead. If the deployed URL ever changes, the new URL needs to be in Supabase's Authentication → URL Configuration → Redirect URLs allowlist, or `resetPasswordForEmail`'s `redirectTo` gets rejected/ignored.

The app is a sidebar SPA with five pages (`.page` sections, toggled by `showPage(name)`), in sidebar order: **Relatório mensal**, **Lembretes**, **Contas**, **Custo fixo por pedido**, **Precificação de produtos** (the sidebar order doesn't match the order the pages appear in the HTML source — don't assume one from the other). `showPage` swaps the visible `.page`, updates the sidebar active state and top bar title, and calls that page's own render function — there is no shared render loop across pages. On narrow viewports the sidebar becomes an off-canvas drawer (`.sidebar.open`, toggled by the `#btnMenu` hamburger button plus a `#sidebarBackdrop` click-to-close overlay) instead of a static column.

### Data layer (one Supabase row per user)

All seven in-memory state values (`data`, `faturamento`, `lembretes`, `lembretesStatus`, `custoFixoConfig`, `produtos`, `configPrecificacao`) map to columns on a single row in the `financeiro_dados` table, keyed by `user_id` (the Supabase Auth user's id):

| Column | Shape | Used by |
|---|---|---|
| `contas` | array of `{ id, descricao, valor, vencimento, categoria, status: 'pendente'\|'pago', tipo: 'despesa'\|'receita', pagoEm }` | Contas, Relatório |
| `faturamento` | object `{ 'YYYY-MM': number }` — manual monthly revenue entry | Relatório |
| `lembretes` | array of `{ id, dia: 1-31, descricao }` — recurring reminder definitions, seeded via `lembretesPadrao()` (contabilidade 10, ERP 10, imposto 20, INSS 20) the first time a user's row is created | Lembretes |
| `lembretes_status` | object `{ 'YYYY-MM': { [lembreteId]: boolean } }` — per-month checklist state | Lembretes |
| `custo_fixo` | object `{ custoFixo: number, pedidos: number }` — last-used values for the fixed-cost-per-order calculator (not month-keyed; just remembers the last inputs) | Custo fixo por pedido |
| `produtos` | array of `{ id, nome, foto, marketplace, custoProduto, taxaFixaMarketplace, comissaoMarketplace, freteMarketplace, custoAds, outrosCustos: [{descricao, valor}], lucroDesejado, precoVenda }` — no per-product `imposto`/`custoEmbalagem`/`custoFixoOperacao`/`comissaoAfiliado`, those come from `configPrecificacao` for every product | Precificação de produtos |
| `config_precificacao` | object `{ imposto, custoEmbalagem, custoFixo, comissaoAfiliado }` — company-wide pricing defaults, filled in once on the "Configurações padrão da precificação" card above the quick calculator; `imposto`/`custoEmbalagem`/`custoFixo` apply to every marketplace and every product with no per-item override | Precificação de produtos |

`carregarDados(userId)` fetches the row on login (`maybeSingle()`); if none exists yet it inserts a fresh default row. Every mutation calls one of `saveContas()`/`saveFaturamento()`/`saveLembretes()`/`saveLembretesStatus()`/`saveCustoFixoConfig()`/`saveProdutos()`/`saveConfigPrecificacao()` — all seven are thin wrappers around a single **`persist()`** that `upsert`s the *entire* row (all columns at once) rather than patching individual fields. This is simple but means concurrent edits from two open tabs/devices will clobber each other's most recent unsaved change — acceptable for a single-user internal tool, worth revisiting if multiple people start editing simultaneously.

Reminders are definitions, not dated events: the same `lembretes` list is checked off against a fresh key each calendar month (`currentYm()`), so checkboxes reset automatically without any migration logic.

`setSyncStatus('saving'|'ok'|'error')` drives the small dot+label in the top bar (`#syncDot`/`#syncText`) so a failed `persist()` (e.g. offline) is visible instead of silently losing the edit.

### Page-specific render functions

- **`renderContas()`** — filters `data` by the active tab (`currentFilter`) and search box, sorts by `vencimento`, rebuilds `#list`, then calls `renderSummaryContas()` for the four dashboard cards. Called after every conta mutation (add/edit/delete/mark-paid).
- **`isVencida(item)`** is the single source of truth for "overdue" (status `pendente` + `vencimento` before today) — used for badges and to exclude overdue items from the "a pagar" bucket (counted separately under "vencidas").
- **`renderRelatorio()`** is the **Mensal** half of the Relatório page (`#relatorioMensalView`, toggled against `#relatorioAnualView` by the `#relatorioTabs` Mensal/Anual tabs — `relatorioView` tracks which is active). It reads `relatorioYm` (state behind the custom month-picker dropdown described below, defaults to current month), sums receitas/despesas pagas whose `pagoEm` falls in that month, and combines with the manual `faturamento[ym]` value: `saldo = faturamento + entrou - saiu`. Editing faturamento toggles `#fatEditRow` inline rather than opening a modal. It also calls **`renderFaturamentoChart(ym)`**, which draws a 12-month rolling bar chart (pure CSS/divs, no chart library) of `faturamento` values ending at the current real month; clicking a bar calls `setRelatorioYm(ym)` to jump the picker to that month and re-render the whole report. The month picker itself (`#btnMesPicker`/`#mesPickerDropdown`) is a custom-built dropdown (year nav + month grid), not a native `<input type="month">`, so it can be styled consistently with the rest of the app — `relatorioMesInput` is kept only as a `{ get value() }` shim for any older code path expecting that shape.
- **`renderRelatorioAnual()`** is the **Anual** half — nav'd by `relatorioAno` (a plain year, via `#anoPickerPrev`/`#anoPickerNext`), it sums `faturamento[ym]` across all 12 months of that year plus receitas/despesas pagas whose `pagoEm` starts with that year (`.slice(0,4)`), and draws a Jan–Dez bar chart of that year specifically (not a rolling 12 months like the mensal chart). Clicking a bar switches back to the Mensal tab and jumps `setRelatorioYm` to that month. `showPage('relatorio')` renders whichever sub-view is/was active so returning to the page after navigating away doesn't show stale numbers.
- **`renderLembretes()`** sorts reminder definitions by `dia`, cross-references `lembretesStatus[currentYm()]` for checked state, and flags a reminder `vencido` when its `dia` has passed this month and it's still unchecked.
- **`renderCustoFixo()`** populates the two calculator inputs from `custoFixoConfig` and calls `calcularCustoFixo()`, which just divides custo fixo by quantidade de pedidos (0 if pedidos is 0) — recalculated live on every keystroke (`input` event) but only persisted on `change` (blur/enter), to avoid an upsert per keystroke. `btnPuxarFixas` shortcuts filling custo fixo from the sum of this month's `categoria === 'Fixas'` despesas that are already marked `pago` — it's a convenience prefill, not a live link, so it won't update automatically if those contas change afterward.
- **`renderProdutos()`** lists `produtos` as cards, each with a thumbnail (`.produto-thumb`, or a `.produto-thumb-placeholder` circle with the product's first initial if no photo), showing preço de venda, margem de contribuição, lucro, and preço recomendado (all recomputed from stored values on every render, not cached). Every card also has an inline **"Ajustar taxas"** toggle (`produtoTaxasAbertoId` tracks which card, if any, has its panel open) — a `.produto-taxas-edit` block letting you change marketplace + taxa fixa + comissão marketplace (+ the ML anúncio/frete subtabs, + a frete field when marketplace is TikTok) without opening the full edit modal; "Salvar" writes straight to that produto's fields and calls `saveProdutos()`. Product photos go through `lerImagemComoDataUrlRedimensionada(file)` (client-side, canvas-based: resizes to a 300px max side and re-encodes as JPEG) before being stored as a data URL on `produto.foto` — this keeps them small since `persist()` re-uploads every product's photo on *every* save, not just the edited product's. Pricing math lives in three pure functions shared by the saved-product list, the product modal preview, *and* the standalone quick calculator described below: `custoFixoTotalProduto(p)` sums every flat R$ cost — `p.custoProduto`, `p.taxaFixaMarketplace`, `p.custoAds`, `p.outrosCustos`, plus the **global** `configPrecificacao.custoFixo`/`configPrecificacao.custoEmbalagem` (not per-product fields — every product and marketplace uses the same custo fixo/embalagem); `percentualDescontosProduto(p)` sums `p.comissaoMarketplace` + `p.freteMarketplace` + the global `configPrecificacao.comissaoAfiliado`/`configPrecificacao.imposto`; `precoRecomendadoProduto(p)` solves the reverse-markup formula `custoFixo / (1 - (%descontos + %lucroDesejado)/100)` — since commissions, imposto, and the profit target are all percentages *of the sale price*, not of cost, isolating price this way is required (a naive `custo * (1 + margem)` markup would undercharge). Returns `null` if the percentages sum to ≥100% (undefined/negative price), which the UI surfaces as a warning instead of a bogus number. `resultadoProduto(p)` computes actual lucro/margem at whatever `precoVenda` is currently set, independent of the recommended price — the two numbers intentionally diverge so the user can compare "what I actually charge" against "what hits my profit target." Because imposto/custo fixo/embalagem/comissão do afiliado are all read from `configPrecificacao` live rather than snapshotted per product, changing any of them on the settings card immediately changes every saved product's displayed numbers too, not just new ones.
- **Shared form-reading helpers**: the product modal (id prefix `p`, e.g. `pCustoProduto`) and every quick-calculator panel (id prefix `qShopee`/`qMl`/`qShein`/`qTiktok`, e.g. `qShopeeCustoProduto`) use identical field-name suffixes, so `lerCampos(prefix, outrosContainerId)` reads any of them generically via `document.getElementById(prefix + suffix)` (falling back to `0` if that prefix+suffix element doesn't exist — e.g. `FreteMarketplace` only exists on the TikTok panel). It deliberately does **not** read `CustoFixoOperacao`/`Imposto`/`CustoEmbalagem`/`ComissaoAfiliado` anymore — none of those have a per-prefix input at all since `custoFixoTotalProduto`/`percentualDescontosProduto` pull them straight from `configPrecificacao`. `atualizarPreview(campos, prefix)` writes that same prefix's result elements (`{prefix}PrecoRecomendado/Lucro/Aviso`, plus `{prefix}Margem` only where that element still exists). `lerCamposProduto()` just wraps `lerCampos('p', 'outrosCustosList')` and adds `nome`/`marketplace`/`foto`. `outrosCustos` rows are plain DOM elements (`.outro-custo-row`) added/removed via `addOutroCustoRow(containerId, ...)`/a delegated `.oc-remove` click handler, then read back with `lerOutrosCustosDoForm(containerId)` at preview-time and submit-time — there's no parallel JS array to keep in sync, the DOM *is* the state while a form is open. Don't let a prefix's field suffixes drift out of sync with the others (e.g. adding a field to the modal but not to `gerarPainelQuickCalc`) or `lerCampos` silently reads `NaN`/0 for the missing one.
- The product modal (`openModalProduto`/`closeModalProduto`, `editingProdutoId` tracks create-vs-edit like the other modals) has no per-product prefill left from `configPrecificacao` — imposto/custo fixo/embalagem/comissão do afiliado aren't stored per product at all, they're always read live from the global config. It also has a visible "✕" (`#btnFecharProduto`) instead of close-on-backdrop-click — clicking the dimmed overlay does nothing, only the ✕ or "Cancelar" close it (a deliberate UX choice after users accidentally closed it with stray clicks).
- **`configPrecificacao`** (`{ imposto, custoEmbalagem, custoFixo, comissaoAfiliado }`) holds the company-wide pricing defaults set once on the "Configurações padrão da precificação" card above the quick calculator (`renderConfigPrecificacao()` populates the four inputs; `salvarConfigPrecificacaoDoForm()` reads them back on `change`, persists via `saveConfigPrecificacao()`, and calls `aplicarConfigPrecificacaoQuickCalc()`). None of the four fields have any DOM representation elsewhere — every calculation reads `configPrecificacao` directly, so there's nothing per-marketplace to keep in sync or that could get wiped when switching tabs. `aplicarConfigPrecificacaoQuickCalc()` just re-runs `atualizarPreview` on all 4 quick-calc panels so their displayed preço recomendado/lucro reflect the new values immediately. `btnPuxarCustoFixoConfig` (next to `cfgCustoFixo`) fills it from `custoFixoPorPedido()` (`custoFixoConfig.custoFixo / custoFixoConfig.pedidos`) as a shortcut, then immediately saves — a convenience prefill, not a live link.
- **`MARKETPLACE_PRESETS`** is the single source of truth for each marketplace's taxa fixa + comissão marketplace (Shopee 18%+R$4, Shein 18%+R$5, Mercado Livre default 14%+R$13.85 — see the ML-specific presets below, and two internal TikTok Shop fee tiers — `tiktok_ate50` 10%+R$4+6% frete / `tiktok_acima50` 6%+R$6+6% frete — since TikTok's commission/fixed-fee actually changes above/below R$50, and it charges freight as a separate % from commission). `aplicarPreset(prefix, presetId)` looks up a preset and writes `{prefix}TaxaFixa`/`{prefix}ComissaoMarketplace`/`{prefix}FreteMarketplace` (only if that field exists, i.e. TikTok) for whichever form (modal, quick-calc panel, or the "Ajustar taxas" card panel) owns that prefix, then calls `marcarSubtabsAtivas(prefix)` — every marketplace-select consumer calls this same function, so they can never drift to different numbers for the same marketplace. `marketplaceLabel(presetId)`/`marketplaceBadgeClass(presetId)` both special-case any id starting with `tiktok` (covers the literal `'tiktok'` value stored on a produto as well as the two internal tier ids) so both TikTok tiers always display as the single generic "TikTok Shop" label/color — the tier distinction is invisible to the user, it only exists internally to pick the right fee numbers. These are reference values the user is expected to adjust (there's a notice in the UI) — marketplace fee structures vary by category/seller plan and change over time, so don't treat them as guaranteed-current.
- **Mercado Livre has two independent quick-pick toggles** inside "Configurar taxas do marketplace", generated by `gerarSubtabsMl(prefix)` (used by the modal's `#pMlConfigWrap`, each quick-calc ML panel, and the saved-product card's `${pt}MlConfigWrap`): **tipo de anúncio** (`ML_ANUNCIO_PRESETS` — Clássico 14% / Premium 19% — writes `{prefix}ComissaoMarketplace`) and **frete** (`ML_FRETE_PRESETS` — Frete grátis R$13.85 / Frete por conta do comprador R$7.95 — writes `{prefix}TaxaFixa`). Clicking one just fills the corresponding number field (still freely editable afterward); `marcarSubtabsAtivas(prefix)` re-derives which button (if any) is `.active` by comparing each `.subtab`'s `data-valor` against its target field's *current* value — so there's no separate "which option is selected" state to keep in sync, editing the number by hand simply un-highlights every button.
- **TikTok only exposes one user-facing option ("TikTok Shop") — the fee tier is always derived from price, never chosen.** The product modal's `#pMarketplace` select has a single `value="tiktok"` option (no `_ate50`/`_acima50` variants in the UI); `faixaTiktokPeloPreco(preco)` returns `tiktok_acima50` if `preco > 50` else `tiktok_ate50`, and that's what actually gets looked up in `MARKETPLACE_PRESETS` for the real taxa/comissão numbers. In the quick-calc TikTok panel (`data-tiktok="true"` on its `.quick-panel`), typing in that panel's `PrecoVenda` field re-applies the matching preset live (in the delegated `input` handler). In the product modal, `ajustarFaixaTiktokModal()` does the same whenever `#pMarketplace` is `'tiktok'` — it unconditionally re-derives the tier from whatever price is currently in the field (no need to preserve a "manually chosen" tier, since there's no such choice anymore). Both "usar preço recomendado" handlers (quick-calc and modal) do a **two-step convergence**: compute the recommended price under the current tier, apply it, re-derive the tier from that price (which may now differ), and if it changed, recompute the price once more under the corrected tier — this sidesteps the chicken-and-egg problem of the fee depending on the very price being calculated. It's a single extra pass, not a loop, since there are only two tiers to converge between. `openModalProduto()` normalizes any produto saved before this change (`marketplace` stored as `'tiktok_ate50'`/`'tiktok_acima50'`) back to the generic `'tiktok'` select value via `.startsWith('tiktok')` — the select has no option matching the old tier-specific values anymore, so setting them directly would silently fail to select anything.
- **Quick calculator** (`#page-precificacao`'s `.chart-card` above the produtos list) has one tabbed panel per marketplace (`MARKETPLACES_QUICK`: Shopee, Mercado Livre, Shein, TikTok Shop — each entry just references a `MARKETPLACE_PRESETS` id, it doesn't duplicate the numbers; TikTok's entry also carries `ehTiktok: true`, ML's carries `ehMl: true`), each panel with its own independent set of fields (custo do produto, comissões, outros custos, etc. — nothing is shared between tabs *except* the global `configPrecificacao` values, which have no per-tab input at all) built once by `initQuickCalcPanels()` from the `gerarPainelQuickCalc(prefix, ehTiktok, ehMl)` template, since generating 4 near-identical forms by hand invites copy-paste drift. Clicking a top-level marketplace tab (`#quickMarketplaceTabs`) just toggles which `.quick-panel` has the `active` class — all 4 panels' DOM and values persist underneath, so switching tabs never loses what you typed. Each panel's inputs/buttons are wired through **one pair of delegated listeners** on `#quickCalcPanels` (`input` for live recalculation plus the TikTok tier check and the ML subtabs, `click` for add/remove-outro-custo, the ML/TikTok subtabs, and "usar preço recomendado") that resolve which panel/prefix triggered the event via `e.target.closest('.quick-panel').dataset.prefix` — this is what makes one code path work for all 4 tabs instead of one listener set per marketplace. `initQuickCalcDefaults()` (called once from `afterLogin`, after `carregarDados` so `configPrecificacao` is loaded) just runs `aplicarPreset(m.prefix, m.preset)` and sets a flat meta-lucro default of 10 per panel — imposto/custo fixo/embalagem/comissão afiliado need no seeding at all since they're read live from `configPrecificacao`, not from any per-panel field. All quick-calc state is session-only (resets on reload, not part of `persist()`) — it's a scratchpad, not a saved record.
- **Both the product modal and every quick-calc panel split their fields into a simple always-visible view and a collapsible `<details class="config-taxas">` "Configurar taxas do marketplace" section** (styled as a bordered button-like `<summary>`, not a plain link, so it doesn't read as decorative). Always visible: custo do produto, custo de ads, meta de lucro, preço recomendado/venda, lucro. Inside the collapsible section: taxa fixa marketplace, comissão marketplace, the ML anúncio/frete subtabs or the TikTok frete field (whichever applies), and outros custos. Nothing pricing-critical is hidden by default *and* editable in only one place — the collapsed section is purely about decluttering the common path, every field inside it still round-trips through the same `lerCampos`/`aplicarPreset` machinery as everything else.

### Shared conventions

- Modal/form pairs (`openModal`/`closeModal` for contas, `openModalLembrete`/`closeModalLembrete` for lembretes, `openModalProduto`/`closeModalProduto` for produtos) are reused for both create and edit; `editingId`/`editingLembreteId`/`editingProdutoId` track which mode is active.
- **Backup/restore** (`btnExport`/`btnImport`) bundles all state stores into one `.json` (currently `{version:5, contas, faturamento, lembretes, lembretesStatus, custoFixoConfig, produtos, configPrecificacao}`) and calls `persist()` once on import. Import also accepts a legacy v1 backup (bare array = contas only) for backwards compatibility. This remains a useful safety net even with cloud storage — keep every store in the export/import pair if you add a new one.

## Conventions

- Content is in Brazilian Portuguese.
- Currency formatted via `Intl`/`toLocaleString('pt-BR', {style:'currency', currency:'BRL'})` — always use `fmt()`, don't hand-roll formatting.
- Dates are stored as `YYYY-MM-DD` strings (native `<input type="date">` format) so they sort correctly with plain string comparison; `formatDate()` converts to `DD/MM/YYYY` only for display.
- CSS custom properties in `:root` mirror the token names used in the sibling `nexus-finance-completo` project (`--navy`, `--accent`, `--green`, `--red`, `--bg`) for visual consistency across the two finance tools — keep them in sync if retheming either.
