# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

**Stellar Controle Financeiro** is a single-file financial tracker for the business's own accounts payable and expenses ("contas a pagar" / "contas que saíram"). It is a private internal tool (not a public SaaS) but is backed by Supabase for auth + storage so the same data is available from any device after logging in — no per-browser `localStorage` lock-in.

## Running / Previewing

Open `index.html` directly in a browser. No server, build step, or package manager. Requires network access to reach Supabase (both for auth and data).

## Deployment

Drag the whole folder into Netlify (app.netlify.com) — pure static HTML, no build command needed. The Supabase URL/anon key are hardcoded in `index.html` (see below) — no environment variables or build-time injection.

## Supabase setup (one-time, per project)

Run `supabase-setup.sql` in the Supabase SQL Editor to create the `financeiro_dados` table and its Row Level Security policies before first use. See `README.md` for the full first-run checklist (creating the login user, email confirmation). That file's header comments also list `alter table` statements for columns (`custo_fixo`, `produtos`) added after the table's initial creation — run those against any pre-existing table that predates a given column.

**Never put the `service_role` key in this file or anywhere client-side** — only the `anon` key belongs in `index.html`. Security comes entirely from Supabase Auth + the RLS policies (`auth.uid() = user_id`) in `supabase-setup.sql`, not from hiding the anon key.

## Architecture

Single file (`index.html`) containing markup, `<style>`, and `<script>`. No frameworks; the only external dependency is `@supabase/supabase-js@2` loaded via CDN `<script>` tag.

The app has two top-level `.screen` divs toggled by `showScreen(id)`: **`screen-login`** (email/senha form, doubles as signup via the "Criar conta" toggle) and **`screen-app`** (the sidebar SPA described below). On load, `initAuth()` checks `sb.auth.getSession()` — an existing session (persisted by supabase-js in its own `localStorage` entry) skips straight to the app; otherwise the login screen shows. `afterLogin(user)` sets `currentUserId`, calls `carregarDados(user.id)` to pull this user's row, then shows the app.

The app is a sidebar SPA with six pages (`.page` sections, toggled by `showPage(name)`): **Contas**, **Relatório mensal**, **Despesas por categoria**, **Lembretes**, **Custo fixo por pedido**, **Precificação de produtos**. `showPage` swaps the visible `.page`, updates the sidebar active state and top bar title, and calls that page's own render function — there is no shared render loop across pages. On narrow viewports the sidebar becomes an off-canvas drawer (`.sidebar.open`, toggled by the `#btnMenu` hamburger button plus a `#sidebarBackdrop` click-to-close overlay) instead of a static column.

### Data layer (one Supabase row per user)

All six in-memory state values (`data`, `faturamento`, `lembretes`, `lembretesStatus`, `custoFixoConfig`, `produtos`) map to columns on a single row in the `financeiro_dados` table, keyed by `user_id` (the Supabase Auth user's id):

| Column | Shape | Used by |
|---|---|---|
| `contas` | array of `{ id, descricao, valor, vencimento, categoria, status: 'pendente'\|'pago', tipo: 'despesa'\|'receita', pagoEm }` | Contas, Relatório, Categorias |
| `faturamento` | object `{ 'YYYY-MM': number }` — manual monthly revenue entry | Relatório |
| `lembretes` | array of `{ id, dia: 1-31, descricao }` — recurring reminder definitions, seeded via `lembretesPadrao()` (contabilidade 10, ERP 10, imposto 20, INSS 20) the first time a user's row is created | Lembretes |
| `lembretes_status` | object `{ 'YYYY-MM': { [lembreteId]: boolean } }` — per-month checklist state | Lembretes |
| `custo_fixo` | object `{ custoFixo: number, pedidos: number }` — last-used values for the fixed-cost-per-order calculator (not month-keyed; just remembers the last inputs) | Custo fixo por pedido |
| `produtos` | array of `{ id, nome, custoProduto, custoFixoOperacao, taxaFixaMarketplace, comissaoMarketplace, comissaoAfiliado, custoEmbalagem, custoAds, outrosCustos: [{descricao, valor}], lucroDesejado, precoVenda }` | Precificação de produtos |

`carregarDados(userId)` fetches the row on login (`maybeSingle()`); if none exists yet it inserts a fresh default row. Every mutation calls one of `saveContas()`/`saveFaturamento()`/`saveLembretes()`/`saveLembretesStatus()`/`saveCustoFixoConfig()`/`saveProdutos()` — all six are thin wrappers around a single **`persist()`** that `upsert`s the *entire* row (all columns at once) rather than patching individual fields. This is simple but means concurrent edits from two open tabs/devices will clobber each other's most recent unsaved change — acceptable for a single-user internal tool, worth revisiting if multiple people start editing simultaneously.

Reminders are definitions, not dated events: the same `lembretes` list is checked off against a fresh key each calendar month (`currentYm()`), so checkboxes reset automatically without any migration logic.

`setSyncStatus('saving'|'ok'|'error')` drives the small dot+label in the top bar (`#syncDot`/`#syncText`) so a failed `persist()` (e.g. offline) is visible instead of silently losing the edit.

### Page-specific render functions

- **`renderContas()`** — filters `data` by the active tab (`currentFilter`) and search box, sorts by `vencimento`, rebuilds `#list`, then calls `renderSummaryContas()` for the four dashboard cards. Called after every conta mutation (add/edit/delete/mark-paid).
- **`isVencida(item)`** is the single source of truth for "overdue" (status `pendente` + `vencimento` before today) — used for badges and to exclude overdue items from the "a pagar" bucket (counted separately under "vencidas").
- **`renderRelatorio()`** reads `relatorioYm` (state behind the custom month-picker dropdown described below, defaults to current month), sums receitas/despesas pagas whose `pagoEm` falls in that month, and combines with the manual `faturamento[ym]` value: `saldo = faturamento + entrou - saiu`. Editing faturamento toggles `#fatEditRow` inline rather than opening a modal. It also calls **`renderFaturamentoChart(ym)`**, which draws a 12-month rolling bar chart (pure CSS/divs, no chart library) of `faturamento` values ending at the current real month; clicking a bar calls `setRelatorioYm(ym)` to jump the picker to that month and re-render the whole report. The month picker itself (`#btnMesPicker`/`#mesPickerDropdown`) is a custom-built dropdown (year nav + month grid), not a native `<input type="month">`, so it can be styled consistently with the rest of the app — `relatorioMesInput` is kept only as a `{ get value() }` shim for any older code path expecting that shape.
- **`renderCategorias()`** groups `despesa`-type contas by `categoria` (filtered by the todas/pago/pendente tab) and renders a proportional CSS bar per category, scaled against the largest category total.
- **`renderLembretes()`** sorts reminder definitions by `dia`, cross-references `lembretesStatus[currentYm()]` for checked state, and flags a reminder `vencido` when its `dia` has passed this month and it's still unchecked.
- **`renderCustoFixo()`** populates the two calculator inputs from `custoFixoConfig` and calls `calcularCustoFixo()`, which just divides custo fixo by quantidade de pedidos (0 if pedidos is 0) — recalculated live on every keystroke (`input` event) but only persisted on `change` (blur/enter), to avoid an upsert per keystroke. `btnPuxarFixas` shortcuts filling custo fixo from the sum of this month's `categoria === 'Fixas'` despesas that are already marked `pago` — it's a convenience prefill, not a live link, so it won't update automatically if those contas change afterward.
- **`renderProdutos()`** lists `produtos` as cards showing preço de venda, margem de contribuição, lucro, and preço recomendado (all recomputed from stored values on every render, not cached). Pricing math lives in three pure functions shared by the saved-product list, the product modal preview, *and* the standalone quick calculator described below: `custoFixoTotalProduto(p)` sums every cost that's a flat R$ amount (produto, operação, taxa fixa, embalagem, ads, outrosCustos); `percentualDescontosProduto(p)` sums the three % deductions (comissão marketplace, comissão afiliado, imposto); `precoRecomendadoProduto(p)` solves the reverse-markup formula `custoFixo / (1 - (%descontos + %lucroDesejado)/100)` — since commissions, imposto, and the profit target are all percentages *of the sale price*, not of cost, isolating price this way is required (a naive `custo * (1 + margem)` markup would undercharge). Returns `null` if the percentages sum to ≥100% (undefined/negative price), which the UI surfaces as a warning instead of a bogus number. `resultadoProduto(p)` computes actual lucro/margem at whatever `precoVenda` is currently set, independent of the recommended price — the two numbers intentionally diverge so the user can compare "what I actually charge" against "what hits my profit target."
- **Shared form-reading helpers**: the product modal (id prefix `p`, e.g. `pCustoProduto`) and every quick-calculator panel (id prefix `qShopee`/`qMl`/`qShein`/`qTiktok`, e.g. `qShopeeCustoProduto`) use identical field-name suffixes, so `lerCampos(prefix, outrosContainerId)` reads any of them generically via `document.getElementById(prefix + suffix)`, and `atualizarPreview(campos, prefix)` writes that same prefix's three result elements (`{prefix}PrecoRecomendado/Margem/Lucro/Aviso`). `lerCamposProduto()` just wraps `lerCampos('p', 'outrosCustosList')` and adds `nome` (the only field the quick calc doesn't have, since it never gets saved). `outrosCustos` rows are plain DOM elements (`.outro-custo-row`) added/removed via `addOutroCustoRow(containerId, ...)`/a delegated `.oc-remove` click handler, then read back with `lerOutrosCustosDoForm(containerId)` at preview-time and submit-time — there's no parallel JS array to keep in sync, the DOM *is* the state while a form is open. Don't let a prefix's field suffixes drift out of sync with the others (e.g. adding a field to the modal but not to `gerarPainelQuickCalc`) or `lerCampos` silently reads `NaN`/0 for the missing one.
- The product modal (`openModalProduto`/`closeModalProduto`, `editingProdutoId` tracks create-vs-edit like the other modals) prefills `custoFixoOperacao` from `custoFixoPorPedido()` (`custoFixoConfig.custoFixo / custoFixoConfig.pedidos`) only for *new* products — editing an existing product leaves its stored value alone, so per-product overrides survive later changes to the global custo-fixo calculator. It also has a visible "✕" (`#btnFecharProduto`) instead of close-on-backdrop-click — clicking the dimmed overlay does nothing, only the ✕ or "Cancelar" close it (a deliberate UX choice after users accidentally closed it with stray clicks).
- **`MARKETPLACE_PRESETS`** is the single source of truth for each marketplace's taxa fixa + comissão marketplace (Shopee 18%+R$4, Mercado Livre 19%+R$6.75, Shein 18%+R$5, and two internal TikTok Shop fee tiers — `tiktok_ate50` 16%+R$4 / `tiktok_acima50` 12%+R$6 — since TikTok's commission/fixed-fee actually changes above/below R$50). `aplicarPreset(prefix, presetId)` looks up a preset and writes `{prefix}TaxaFixa`/`{prefix}ComissaoMarketplace` for whichever form (modal or quick-calc panel) owns that prefix — both the product modal's `#pMarketplace` select and the quick-calc panels call this same function, so the two calculators can never drift to different numbers for the same marketplace. `marketplaceLabel(presetId)`/`marketplaceBadgeClass(presetId)` both special-case any id starting with `tiktok` (covers the literal `'tiktok'` value stored on a produto as well as the two internal tier ids) so both TikTok tiers always display as the single generic "TikTok Shop" label/color — the tier distinction is invisible to the user, it only exists internally to pick the right fee numbers. These are reference values the user is expected to adjust (there's a notice in the UI) — marketplace fee structures vary by category/seller plan and change over time, so don't treat them as guaranteed-current.
- **TikTok only exposes one user-facing option ("TikTok Shop") — the fee tier is always derived from price, never chosen.** The product modal's `#pMarketplace` select has a single `value="tiktok"` option (no `_ate50`/`_acima50` variants in the UI); `faixaTiktokPeloPreco(preco)` returns `tiktok_acima50` if `preco > 50` else `tiktok_ate50`, and that's what actually gets looked up in `MARKETPLACE_PRESETS` for the real taxa/comissão numbers. In the quick-calc TikTok panel (`data-tiktok="true"` on its `.quick-panel`), typing in that panel's `PrecoVenda` field re-applies the matching preset live (in the delegated `input` handler). In the product modal, `ajustarFaixaTiktokModal()` does the same whenever `#pMarketplace` is `'tiktok'` — it unconditionally re-derives the tier from whatever price is currently in the field (no need to preserve a "manually chosen" tier, since there's no such choice anymore). Both "usar preço recomendado" handlers (quick-calc and modal) do a **two-step convergence**: compute the recommended price under the current tier, apply it, re-derive the tier from that price (which may now differ), and if it changed, recompute the price once more under the corrected tier — this sidesteps the chicken-and-egg problem of the fee depending on the very price being calculated. It's a single extra pass, not a loop, since there are only two tiers to converge between. `openModalProduto()` normalizes any produto saved before this change (`marketplace` stored as `'tiktok_ate50'`/`'tiktok_acima50'`) back to the generic `'tiktok'` select value via `.startsWith('tiktok')` — the select has no option matching the old tier-specific values anymore, so setting them directly would silently fail to select anything.
- **Quick calculator** (`#page-precificacao`'s `.chart-card` above the produtos list) has one tabbed panel per marketplace (`MARKETPLACES_QUICK`: Shopee, Mercado Livre, Shein, TikTok Shop — each entry just references a `MARKETPLACE_PRESETS` id, it doesn't duplicate the numbers; TikTok's entry also carries `ehTiktok: true`), each panel with its own independent set of fields (own custo do produto, comissões, outros custos, etc. — nothing is shared between tabs) built once by `initQuickCalcPanels()` from the `gerarPainelQuickCalc(prefix, ehTiktok)` template, since generating 4 near-identical ~15-field forms by hand invites copy-paste drift. Clicking a top-level marketplace tab (`#quickMarketplaceTabs`) just toggles which `.quick-panel` has the `active` class — all 4 panels' DOM and values persist underneath, so switching tabs never loses what you typed. Each panel's inputs/buttons are wired through **one pair of delegated listeners** on `#quickCalcPanels` (`input` for live recalculation plus the TikTok tier check, `click` for add/remove-outro-custo, "usar preço recomendado", and "puxar custo fixo") that resolve which panel/prefix triggered the event via `e.target.closest('.quick-panel').dataset.prefix` — this is what makes one code path work for all 4 tabs instead of one listener set per marketplace. `initQuickCalcDefaults()` (called once from `afterLogin`, after `carregarDados` so `custoFixoConfig` is loaded) seeds every panel with `custoFixoPorPedido()` plus `aplicarPreset(m.prefix, m.preset)` and flat imposto/comissão afiliado/meta-lucro defaults. All quick-calc state is session-only (resets on reload, not part of `persist()`) — it's a scratchpad, not a saved record.

### Shared conventions

- Modal/form pairs (`openModal`/`closeModal` for contas, `openModalLembrete`/`closeModalLembrete` for lembretes, `openModalProduto`/`closeModalProduto` for produtos) are reused for both create and edit; `editingId`/`editingLembreteId`/`editingProdutoId` track which mode is active.
- **Backup/restore** (`btnExport`/`btnImport`) bundles all state stores into one `.json` (currently `{version:4, contas, faturamento, lembretes, lembretesStatus, custoFixoConfig, produtos}`) and calls `persist()` once on import. Import also accepts a legacy v1 backup (bare array = contas only) for backwards compatibility. This remains a useful safety net even with cloud storage — keep every store in the export/import pair if you add a new one.

## Conventions

- Content is in Brazilian Portuguese.
- Currency formatted via `Intl`/`toLocaleString('pt-BR', {style:'currency', currency:'BRL'})` — always use `fmt()`, don't hand-roll formatting.
- Dates are stored as `YYYY-MM-DD` strings (native `<input type="date">` format) so they sort correctly with plain string comparison; `formatDate()` converts to `DD/MM/YYYY` only for display.
- CSS custom properties in `:root` mirror the token names used in the sibling `nexus-finance-completo` project (`--navy`, `--accent`, `--green`, `--red`, `--bg`) for visual consistency across Stellar/Nexus finance tools — keep them in sync if retheming either.
